    Lab2： BombLab

    查看所有寄存器：layout regs

    调试过程显示汇编代码：display /i $pc

    Phase 1

    解析过程

    Phase 1 作为第一关，其反汇编代码也是最短的，算是小热身，省得一上来把人吓退了（

    先看和炸弹相关的部分：

    0000000000400ee0 <phase_1>:
    ...
    400ee4: be 00 24 40 00          mov    $0x402400,%esi # 这个赋值显然有特殊用意，但先不管
    ...
    400eee: 85 c0                   test   %eax,%eax
    400ef0: 74 05                   je     400ef7 <phase_1+0x17> 
    ; 当eax为0时会跳过函数explode_bomb，即炸弹解除
    400ef2: e8 43 05 00 00          call   40143a <explode_bomb>
    ...

    那么%eax的值又是哪里来的呢？注意到前一行有call函数调用，说明%eax大概率就是这个函数的返回值。

    我们需要知道这个strings_not_equal()在什么情况下会返回 0 （虽然顾名思义很显然就是两字符串相等的时候……），再来看一下它的汇编：

    0000000000401338 <strings_not_equal>:
  ...
  40133c:   48 89 fb                mov    %rdi,%rbx
  40133f:   48 89 f5                mov    %rsi,%rbp 
  401342:   e8 d4 ff ff ff          call   40131b <string_length>
  401347:   41 89 c4                mov    %eax,%r12d
  40134a:   48 89 ef                mov    %rbp,%rdi
  40134d:   e8 c9 ff ff ff          call   40131b <string_length>
  401352:   ba 01 00 00 00          mov    $0x1,%edx
  401357:   41 39 c4                cmp    %eax,%r12d
  40135a:   75 3f                   jne    40139b <strings_not_equal+0x63>
  ; 如果这一步跳转被执行，函数将返回，此时%eax不为0，炸弹爆炸
  ; 所以要求 %eax 必须与 %r12d 相等
  ...

    看到两个string_length就很自然知道，首先比较了两个字符串长度，结合后面代码，两个寄存器的值不相等则爆炸，那么401357处的%eax和%r12d肯定各自对应用户输入的字符串长度和答案字符串的长度。

    要搞清楚对应关系，最快方法还是用一个测试字符串去进行 GDB 调试：

    $ gdb bomb
(gdb) b *0x401357
Breakpoint 1 at 0x401357
(gdb) r
...
which to blow yourself up. Have a nice day!
123 # 测试字符串

Breakpoint 1, 0x0000000000401357 in strings_not_equal ()
(gdb) p $eax
$1 = 52
(gdb) p $r12d
$2 = 3

    显然答案字符串的长度为 52，并且可以确定答案字符串的地址作为参数传递是发生在40134a处。往前推可以发现这个地址来自于%rsi，它是在phase_1中被赋值的，这个值就是0x402400。

    知道了地址，就可以通过查看二进制文件来找出这段字符串了。字符串的存储位置可能在.rodata段、.data段乃至在栈帧上，但考虑到这只是第一关，答案字符串作为常量存储在.rodata段是最有可能的。

    $ objdump -s -j .rodata bomb # 查看.rodata 段的全部内容
...
402400 426f7264 65722072 656c6174 696f6e73  Border relations
402410 20776974 68204361 6e616461 20686176   with Canada hav

    显然答案字符串就是：

    Border relations with Canada have never been better.

    算上一个句号正好 52 个字符。

    Phase 2

    解析过程

    注意到phase_2()函数的反汇编结果中调用了read_six_numbers()函数：

    0000000000400efc <phase_2>:
  400efc:   55                      push   %rbp
  400efd:   53                      push   %rbx
  400efe:   48 83 ec 28             sub    $0x28,%rsp 
  ; 开辟40个字节的栈空间，即5个元素
  400f02:   48 89 e6                mov    %rsp,%rsi
  ; 将栈顶地址保存在%rsi中
  400f05:   e8 52 05 00 00          call   40145c <read_six_numbers>

    猜测读入的字符串由 6 个数字构成。

    分析read_six_numbers()的反汇编代码，可以看到有一行sscanf()的调用，其返回值为成功赋值的字段个数，后面紧接着将返回值与0x5进行比较，若返回值不大于 5，将会执行call 40143a <explode_bomb>，炸弹爆炸。所以该函数的功能就是判断输入字符串是否包含6个数字，否则爆炸。

    继续看phase_2()：

    首先注意到调用返回之后发生了与(%rsp)的比较：

    400f05: e8 52 05 00 00          call   40145c <read_six_numbers>
400f0a: 83 3c 24 01             cmpl   $0x1,(%rsp)
400f0e: 74 20                   je     400f30 <phase_2+0x34> ;如果(%rsp)不为1，爆炸

    这里要求(%rsp)要等于 1，这应该是六个数字的其中一个，但还不能确定第几个数字会写入(%rsp)。

    如果躲过这次爆炸，会跳转到400f30处：

    400f30: 48 8d 5c 24 04          lea    0x4(%rsp),%rbx ;rbx=rsp+4
400f35: 48 8d 6c 24 18          lea    0x18(%rsp),%rbp ;rbp=rsp+24

    修改完之后回到400f17处：

    400f17: 8b 43 fc                mov    -0x4(%rbx),%eax 
; 将%rbx寄存器的值-4，并视为一个地址（其实是%rsp），从中读取数据赋给%eax。等价于mov (%rsp),%eax
; 注意，上面的-4操作并没有实际修改%rbx的值，rbx仍然是rsp+4
400f1a: 01 c0                   add    %eax,%eax ; %eax*2
400f1c: 39 03                   cmp    %eax,(%rbx)
400f1e: 74 05                   je     400f25 <phase_2+0x29> ;如果%eax不等于(%rbx)，爆炸

    这段代码总体的意图是，要求(%rsp)的2倍等于(%rbx)。前面可知(%rsp)为1，则(%rbx)需要为2。注意到%rbx恰为%rsp+4，即上述两个整数 1 和 2 的存储地址实际上是相邻的。

    如果躲过这次爆炸，会跳转至400f25处：

    400f25: 48 83 c3 04             add    $0x4,%rbx ; rbx=rsp+8
400f29: 48 39 eb                cmp    %rbp,%rbx ; rbp=rsp+24
400f2c: 75 e9                   jne    400f17 <phase_2+0x1b>

    这里可以看出来，400f17和400f25两处的代码共同构成了一个for循环，在这个循环中，%rbx的初始值为%rsp+4，并在循环末尾递增4，直到%rbx == %rsp+24时结束。即相当于以下 C 语言代码：

    a[0] = 1; // (%rsp)为1
for(int i = 1;i < 6;i++){
    if(a[i - 1] + a[i - 1] != a[i])
        explode_bomb(); //爆炸
}

    那么这六个数字中除第一个数以外，其他每一个数都应该是前一个数的 2 倍，因此答案字符串为：

    1 2 4 8 16 32

    调试技巧

    看汇编代码可以很快发现上面所说的有一段来回倒腾的疑似循环的过程，且地址与%rsp相关。又已知输入字符串是六个整数，要猜测输入的整数和地址的关系，其实可以把断点打在400f0a: 83 3c 24 01 cmpl $0x1,(%rsp)处，然后查看从%rsp开始的连续 6 个字长的内存地址中的值：

    $ gdb bomb
(gdb) b *0x400f0a
Breakpoint 1 at 0x400f0a
(gdb) r
...
Phase 1 defused. How about the next one?
1 2 3 4 5 6 # 测试输入

Breakpoint 1, 0x0000000000400f0a in phase_2 ()
(gdb) p $rsp # 查看%rsp存储的地址
$1 = (void *) 0x7fffffffdc50
(gdb) x/6wd 0x7fffffffdc50 # x/6wd 中的w表示以4字节为一个单位，打印六个单位的值，d表示以十进制格式输出值
0x7fffffffdc50: 1       2       3       4
0x7fffffffdc60: 5       6

    可以看到我们的输入被依次存放在了内存地址%rsp、%rsp+4、%rsp+8……%rsp+20处。

    Phase 3

    解析过程

    先看爆炸条件：

    0000000000400f43 <phase_3>:
  ...
  400f47:   48 8d 4c 24 0c          lea    0xc(%rsp),%rcx ;rcx=rsp+12
  400f4c:   48 8d 54 24 08          lea    0x8(%rsp),%rdx ;rdx=rsp+8
  400f5b:   e8 90 fc ff ff          call   400bf0 <__isoc99_sscanf@plt>
  400f60:   83 f8 01                cmp    $0x1,%eax
  400f63:   7f 05                   jg     400f6a <phase_3+0x27>
  400f65:   e8 d0 04 00 00          call   40143a <explode_bomb>
  ; 如果sscanf赋值的字段数<=1，爆炸
  400f6a:   83 7c 24 08 07          cmpl   $0x7,0x8(%rsp)
  400f6f:   77 3c                   ja     400fad <phase_3+0x6a>
  ; 如果0x8(%rsp)>7（仅比较低4字节），爆炸
  ...
  400fad:   e8 88 04 00 00          call   40143a <explode_bomb>
  ...
  400fbe:   3b 44 24 0c             cmp    0xc(%rsp),%eax
  400fc2:   74 05                   je     400fc9 <phase_3+0x86>
  400fc4:   e8 71 04 00 00          call   40143a <explode_bomb>
  ; 如果eax≠(rsp+12)，爆炸
  ...

    前 6 条指令是我们在 Phase 2 中打过交道的sscanf，查看.rodata段的0x4025cf处的字符串就能确定输入的格式是什么：

    4025c0 702e0025 64202564 20256420 25642025  p..%d %d %d %d %  
4025d0 64202564 00457272 6f723a20 5072656d  d %d.Error: Prem
# 最后的百分号才是字符串的起始字符

    这样就确定了输入字符串的格式是"%d %d"，又因为sscanf的返回值≤1就爆炸，所以可以确定要输入的就是 2 个int整数。并且这两个整数会被依次保存到rsp+8和rsp+12。

    那么cmpl $0x7,0x8(%rsp)的含义也一并明确了：如果输入的第一个整数的低4字节＞7，爆炸。（结合下面的分析还可以知道，这里其实是为了限制switch跳转表下标的取值，因为总共只有 8 个分支）

    另外注意到phase_3中有连续的 8 组指令具有相似的结构（都是一条mov和一条jmp），这其实是switch的特征。那么一共 8 个case，每一个都是为%eax 赋值后再与我们输入的第二个整数比较（400fbe处的代码）若不相等则爆炸。换句话说，只要确定哪个case会被执行，就能确定第二个整数的值。

    如何确定呢？在这 8 组指令之前，还有两行未分析的代码：

    400f71: 8b 44 24 08             mov    0x8(%rsp),%eax ;将第一个数保存到eax
400f75: ff 24 c5 70 24 40 00    jmp    *0x402470(,%rax,8) ;跳转到(0x402470+8*%rax)处

    这里的jmp特征就比较明显了，其实就是利用switch跳转表进行跳转。跳转表位于0x402470处，而%rax，也就是我们输入的第一个整数，将作为跳转表的下标（补充：跳转表实际上是一个指针数组，而指针大小为 8 字节，所以这里要写8*%rax作为地址偏移量）。

    跳转表也保存在.rodata段，所以查看0x402470处的字符串：

    402470 7c0f4000 00000000 b90f4000 00000000  |.@.......@.....
402480 830f4000 00000000 8a0f4000 00000000  ..@.......@.....
402490 910f4000 00000000 980f4000 00000000  ..@.......@.....
4024a0 9f0f4000 00000000 a60f4000 00000000  ..@.......@.....

    这里有每个下标所对应的跳转地址，且可以看出来是用小端法保存的，如7c0f4000其实是地址400f7c，以此类推。并且可以看出下标和跳转分支地址的对应关系，如下表所示：

    下标 跳转分支地址
    0 400f7c
    1 400fb9
    2 400f83
    3 400f8a
    4 400f91
    5 400f98
    6 400f9f
    7 400fa6

    这里似乎设了个小坑，因为乍眼一看跳转分支地址的顺序差不多就是在汇编代码中的出现顺序，但唯独case 1是个例外。

    进一步地，可以用 GDB 调试验证一下：

    (gdb) b *0x400f75
(gdb) r
...
0  207 #测试输入

Breakpoint 1, 0x0000000000400f75 in phase_3 ()
(gdb) display /i $pc
=> 0x400f75 <phase_3+50>:   jmp    *0x402470(,%rax,8)
(gdb) si
0x400f7c in phase_3()
1: x/i $pc
=> 0x400f7c <phase_3+57>:   mov $0xcf,%eax #eax=207

    由上述结果可以确定，0 207这个输入是正确的。

    我们之前说只要知道跳转到了哪里、确定了%eax的值，就能知道第二个整数应该输入什么。而跳转到哪里实际上是由第一个整数决定的，也就是说以下每行输入其实都是答案：

    0 207
1 311
2 707
3 256
4 389
5 206
6 682
7 327

    源码还原

    我们可以进一步还原出phase_3的源码大致结构：

    void phase_3(char* input){
    int num1, num2, n=0;
    n = sscanf(input, "%d %d", &num1, &num2);
    if(n<=1){
        explode_bomb();
    }
    if(num1>7){
        explode_bomb();
    }
    int x;
    switch(num1):
    case 0:
        x = 207; // mov    $0xcf,%eax
        break;
    case 1: // 注意跳转到了400fb9处的代码
        x = 311; // mov    $0x137,%eax
        break;
    case 2:
        x = 707; // mov    $0x2c3,%eax
        break;
    case 3:
        x = 256; // mov    $0x100,%eax
        break;
    case 4:
        x = 389; // mov    $0x185,%eax
        break;
    case 5:
        x = 206; // mov    $0xce,%eax
        break;
    case 6:
        x = 682; // mov    $0x2aa,%eax
        break;
    case 7:
        x = 327; // mov    $0x147,%eax
        break;
    if(x != num2){ 
        explode_bomb();
    }
    return;
}
